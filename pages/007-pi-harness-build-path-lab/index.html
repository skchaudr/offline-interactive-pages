Pi Harness Build Path

A practical architecture roadmap: how to grow pi-coding-agent into a robust personal coding/automation agent by encoding behavior in deterministic Pi extensions instead of expanding prompt files.

Core Framing: What this is (and isn't)

The Anti-Goals

  • This is not a Claude Code clone. We aren't building a monolithic CLI wrapper.
  • This is not OpenClaw round two. We aren't building a heavy agent framework.
  • Personal context files are just a contract. SOUL.md, USER.md, AGENTS.md, MEMORY.md, and TOOLS.md define operating bounds, not infrastructure.

The Key Inversion

More behavior in extension code, less prompt structure.

Stop trying to make the LLM follow complex control flows via prompts. Instead, write deterministic runtime hooks (extensions) that enforce behavior programmatically.

The Paradigm Shift

Anti-Pattern Prompt Sprawl

Trying to teach the model every rule in AGENTS.md.

// Bloated Prompt Rule "Before you run a command, you must check if it is destructive. If it is destructive, you must ask the user for permission. If the user says no, you must stop. Also, remember to read the context files before doing anything..."
  • High token cost
  • Unreliable compliance
  • Hard to test or debug
Target State Extension Code

Encoding behavior in deterministic API hooks.

// Runtime Hook (Deterministic) export function preToolHook(toolCall) { if (isDestructive(toolCall)) { return requirePermission(toolCall); } return continueExecution(); }
  • Zero prompt bloat
  • 100% reliable execution
  • Standard software testing

Architecture Overview

The Pi extension API mental model is straightforward: wrap the agent's core loop with lifecycle hooks. Instead of asking the agent to remember context, inject it automatically via a Pre-Request hook. Instead of asking the agent to verify destructive commands, catch them in a Pre-Tool hook.

Extension Patterns

Permission Classifier

Intercepts tool calls to classify commands. If rm -rf is detected, pauses execution and prompts the Operator for approval.

Hook: Pre-Tool

Live Repo Context

Automatically scans the repository root for AGENTS.md or similar files and invisibly injects them into the system prompt.

Hook: Pre-Request

Sub-agent Fork

Provides an Explore Tool that lets the main agent spawn a lightweight sub-agent to traverse code and return a summary.

Custom Tool

Multi-model Routing

Dynamically switches LLM providers (e.g., Haiku for exploration, Opus for complex refactoring) based on task parameters.

Hook: Router

GDD Orchestration

Goal-Driven Development integration. Parses .plan files and tracks progress systematically outside of the LLM context window.

Hook: Lifecycle

Build Stages

⚠️ The Skip List

Do not build these features early on. They are distractions:

  • Complex vector databases (use ripgrep/grep first)
  • Custom UI dashboards (CLI is fine)
  • Over-engineered memory systems (write to a markdown file instead)

Stage 0: Operator Contract

Establish the foundation. Create your personal context files without any agent code changes. These are the "operating contract".

# Task Packet: Stage 0
1. Create `USER.md` defining your preferences.
2. Create `SOUL.md` defining the agent's persona.
3. Add a basic `AGENTS.md` to your workspace.
4. Verify the agent can read them manually.

Stage 1: Safety & Permissions (Permission Classifier)

Build the first extension. Intercept bash commands and require operator approval for destructive actions.

# Task Packet: Stage 1
1. Scaffold a Pi extension: `safety-guard`.
2. Implement `preToolHook`.
3. Regex match `rm`, `drop`, `delete`.
4. If matched, prompt user in CLI.
5. Return modified tool call or abort.

Stage 2: Invisible Context (Live Repo Injection)

Stop asking the LLM to "read the README". Inject it via extension automatically.

# Task Packet: Stage 2
1. Create extension `repo-context`.
2. Implement `preRequestHook`.
3. Check for `AGENTS.md` in `process.cwd()`.
4. Append contents to system prompt invisibly.

Stage 3: Sub-agent Fork (Explore Tool)

Give the agent a dedicated tool to spawn a cheaper model to read large codebases.

Stage 4: Multi-model Routing

Optimize costs by routing specific tool calls or tasks to specialized models.

Stage 5: GDD Orchestration Integration

Implement formal state machines for task planning and execution outside the prompt window.

Common Traps

Don't build heavy, generic agent frameworks. Keep extensions small, single-purpose, and tied strictly to your personal workflow needs.

If your AGENTS.md becomes a 500-line script of conditional rules ("If X happens, do Y, unless Z"), you are doing it wrong. Move that logic into a deterministic extension hook.

Claude Code relies heavily on massive prompt structures for behavior. The Pi Build Path aims to invert this: thin prompts, thick extension logic.

LLMs are reasoning engines, not reliable state machines. Do not ask the LLM to manage complex application state or loops. Let the host code (extensions) handle the while-loops and conditional execution paths.

What to build first? (Decision Flow)

Do you frequently run dangerous commands by accident?

👉 Build the Permission Classifier Extension (Stage 1)

Protect your workspace before scaling automation.

Are you tired of telling the agent to "read the README"?

👉 Build the Live Repo Context Injection (Stage 2)

Automate context loading so every session starts smart.

👉 Look into Sub-agent Forks (Stage 3)

If baseline context and safety are good, expand capabilities with multi-agent delegation.
