1. Latency vs Throughput

Latency is the time it takes for one request to complete. Throughput is how many requests can be processed over a given period of time.

Imagine a highway: Latency is the speed limit. Throughput is the number of lanes.

2. Queues and Workers

When clients send requests faster than servers can process them, we need to decouple them. A Queue acts as a buffer. Workers pull from the queue at their own pace.

3. Retries, Backoff, and Idempotency

Network calls fail. When they do, we should retry. But retrying immediately can overwhelm a struggling server, so we use exponential backoff (waiting longer between each try).

Crucially, retries require the operation to be idempotent: doing it multiple times has the same effect as doing it once (e.g., "set x to 5" is idempotent, "add 1 to x" is not).

4. Leader/Follower Replication

To prevent data loss if a database crashes, we replicate data. In Leader/Follower (Primary/Replica), writes go to the Leader, which then streams the changes to Followers. Reads can optionally go to followers to increase read throughput.

If the leader dies, a follower must be promoted to the new leader.

> Cluster ready.

5. Consistency vs Availability (CAP Theorem Intro)

In a distributed system, a network partition (communication break) forces a choice:

  • Consistency: Wait for all nodes to agree (might timeout/fail = unavailable).
  • Availability: Return whatever local data a node has (might be stale = inconsistent).

Many modern systems choose Eventual Consistency: nodes will return stale data to remain available, but will eventually sync up once the network recovers.
